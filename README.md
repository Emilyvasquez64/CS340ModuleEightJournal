* How do you write programs that are maintainable, readable, and adaptable? Especially consider your work on the CRUD Python module from Project One, which you used to connect the dashboard widgets to the database in Project Two. What were the advantages of working in this way? How else could you use this CRUD Python module in the future?
  * When I think about writing programs that are maintainable, readable, and adaptable, I think about structure and clarity first. Code should be organized in a way that someone else can open it and understand what is happening without guessing. In my CRUD Python module from Project One, I focused on keeping everything inside a class with clearly named methods like create() and read(). Each method has a specific job, and the comments explain what the function does, what parameters it takes, and what it returns. That makes the code easier to maintain because if something breaks or needs to be updated, I know exactly where to look.
Working this way had a big advantage when I moved into Project Two and connected the dashboard widgets to the database. Since the CRUD logic was already separated into its own module, I did not have to rewrite database code inside the dashboard. I could just import the AnimalShelter class and call the methods I already built. This saved time and reduced errors because I was reusing tested code instead of starting from scratch. It also made the dashboard cleaner because all of the database connection details stayed in one place.
In the future, I could reuse this CRUD module for other applications that need to connect to the same MongoDB database. For example, if Grazioso Salvare wanted a mobile app or reporting system, I could use the same backend module. I could also expand it by adding update and delete functions, or by improving security features. Because it is modular and object oriented, it can grow with the company’s needs instead of being thrown away.
* How do you approach a problem as a computer scientist? Consider how you approached the database or dashboard requirements that Grazioso Salvare requested. How did your approach to this project differ from previous assignments in other courses? What techniques or strategies would you use in the future to create databases to meet other client requests?
  *When approaching a problem as a computer scientist, I try to break everything down into smaller pieces. With the database and dashboard requirements for Grazioso Salvare, I first thought about what the client actually needed. They wanted to filter animal shelter data to find dogs that meet certain rescue training criteria. So instead of thinking about code right away, I thought about the data. What fields matter most? Breed, age, outcome type, and animal type were important. Once I understood that, designing the database queries became much easier.
This project felt different from assignments in other courses because it felt more realistic. In many past classes, assignments were smaller and focused on one concept at a time. This project required thinking about how multiple parts work together, like the database, backend module, and dashboard interface. It felt more like building something for an actual client instead of just turning in homework.
In the future, I would continue using strategies like planning data structures before coding and creating modular components that can be reused. I would also spend more time thinking about scalability. If a client suddenly has thousands or millions of records, the database design and queries need to handle that efficiently. I would also consider adding indexing and stronger validation rules from the beginning.
